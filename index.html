AI 工作流亮点

运营模型 RFM 指标计算，用户分层，召回策略生成，优惠预算约束，推荐商品匹配，营销文案生成，JSON结构化校验

我的 PM 判断

核心痛点

数据分散，难以串联（客户可能在多渠道出现）；投放粗放（一个物料打所有）；运营低效（人工做不到“千人千面”）

面向企业标书、用户手册、申报书等严谨结构类文档，完成从需求解析到分章节成稿的生成链路。

AI 工作流亮点

核心痛点

营销增长 热点选品

LLM 自动选品发微博

围绕热点新闻自动匹配商品，并生成适合社交平台传播的营销文案。

适用场景

品牌自营商品库、热点借势营销、社媒内容生产。

痛点

人工追热点和选品耗时，热点与商品关联弱时容易产出硬广。

工作流拆解

热点新闻抓取 - 商品关联分析 - 商品库检索 - 平台文案生成 - 风格校验。

我的产品判断

营销场景低风险、高迭代，重点不是“生成更多”，而是提高热点与商品的相关性。

营销增长 RFM 分层

零食店分层营销策略制定

基于 RFM、偏好品类、购买历史和会员等级生成差异化营销策略。

适用场景

零售门店、会员运营、沉睡用户召回、优惠券策略生成。

痛点

不同用户价值和需求不同，统一优惠策略会浪费预算。

工作流拆解

RFM 计算 - 用户分群 - 召回策略匹配 - 优惠预算约束 - 推荐商品匹配 - 召回文案生成 - JSON 校验。

我的产品判断

增长类 Agent 应先接入结构化数据和预算约束，再让 LLM 负责策略表达。

企业效率 法务审核

智能法务合同审核助手

面向小微企业的合同风险初筛与审核意见生成 Agent。

适用场景

合同初筛、主体资质检查、旧法引用审查、风险条款提醒。

痛点

小微企业缺少稳定法务资源，但合同风险又需要被及时发现。

工作流拆解

合同文本读取 - 主体信息提取 - 旧法引用审查 - 风险条款识别 - 审核结论生成。

我的产品判断

法务类是高风险场景，产品定位必须是初筛和辅助，不替代专业法律判断。

企业效率 文档生成

企业级结构化文档生成

以标书撰写为核心场景，验证强结构长文档生成链路。

适用场景

标书、用户手册、申报书、内部制度文档等结构化写作。

痛点

文档内容长、格式强、上下文复杂，一次性生成容易失控。

工作流拆解

文件读取 - 需求提取 - 一级框架生成 - 二级目录扩写 - 正文分章节生成 - 飞书文档写入。

我的产品判断

长文本生成要拆节点、拆章节、保留可编辑结果，而不是追求一次成稿。

情感陪伴 长期记忆

AI Lover 情感陪伴 Agent

通过角色设定、长期记忆检索和写入，模拟追星/二次元陪伴体验。

适用场景

追星人群、二次元角色陪伴、轻量情绪支持、语音包扩展。

痛点

普通陪聊机器人容易人设漂移，也记不住用户偏好和关系历史。

工作流拆解

角色人设 - 长期记忆检索 - 对话状态判断 - 记忆写入 - 情绪化回复生成。

我的产品判断

情感陪伴是低精度、重共情场景，关键是人设一致性、边界感和记忆体验。

学术研究 AI for Code

跨语言代码检索研究

基于 LLM 摘要增强和全局对比学习的跨语言代码检索框架。

适用场景

IDE 智能检索、代码资产复用、跨语言迁移辅助、AI 编程助手召回。

痛点

关键词检索难以识别跨语言语义等价代码，开发者复用成本高。

工作流拆解

代码摘要生成 - 查询/代码 embedding - 跨语言语义对齐 - 相似度检索 - 结果评估。

我的产品判断

AI PM 不必只会搭应用，也要理解 embedding、召回质量和评测指标如何影响产品体验。

学术研究如何转化成产品价值

论文不是只写算法细节，而是证明我能理解模型能力、检索链路和开发者工具场景。

Lost in Translation, Found in Summary: An End-to-End Framework for Cross-Language Code Retrieval

研究问题：当开发者用一种编程语言表达需求时，如何从另一种语言的大量代码中检索到语义等价的实现。方法上通过 LLM 摘要增强代码语义，并用全局对比学习把不同语言的代码对齐到更统一的 embedding 空间。

  • HR 能理解的价值 让“找代码”从关键词匹配变成语义匹配，帮助开发者跨语言复用经验，降低迁移和学习成本。
  • 产品场景 可落到 IDE 智能检索、企业代码资产复用、跨语言迁移辅助、AI 编程助手的上下文召回。
  • 我的差异化优势 不仅会搭 Agent，也理解 LLM 摘要、向量检索、embedding 对齐和评价指标背后的产品影响。

PM Thinking

AI PM 能力结构

适合面向 Agent、AI 工作流、开发者工具、知识管理和效率工具类岗位。

P

问题定义

从用户任务、使用频次、失败成本和替代方案判断 AI 是否值得介入。

W

工作流设计

把复杂任务拆成输入、检索、推理、工具调用、生成、校验和兜底。

M

模型理解

理解 LLM、embedding、RAG、提示词和评测指标对体验的影响。

D

原型验证

用 Coze 和 vibe coding 快速做可试用原型，验证主路径与结果质量。

01 | 需求

先讲清楚用户为什么需要

用痛点、场景、用户路径和失败状态描述需求，不只描述“做了一个 Agent”。

02 | 方案

再讲清楚 AI 如何参与

明确哪些由 LLM 处理，哪些由规则、知识库、工具调用或人工确认完成。

03 | 验证

最后讲清楚如何迭代

用测试样例、错误案例、输出稳定性和用户反馈判断方案是否可继续推进。
